# 개발 레포 핸드오프 — AI 기반 학부모 민원 대응 보조 시스템

> **이 문서 하나로 새 Claude Code 세션이 프로젝트 전체 맥락을 파악하고 곧바로 개발을 이어간다.**
> 자료 레포(`2026-project`, 연구·통계·논문·그래프 원본)에서 추출·통합한 개발용 핸드오프 문서다.
> 작성일 기준 최신 설계(설계 다이어그램 2장 + 폴더 문서 통합). 연구 근거 원본은 자료 레포 참조.

## 0. 한 줄 요약 & 핵심 메시지

학부모 악성 민원으로부터 **교사를 보호**하고, 그 결과 **학생의 학습권을 회복**하는 **로컬 추론 기반 3단계 AI 시스템**.

> 핵심 메시지: **"교사를 보호함으로써 학생의 학습권을 회복한다."**
> AI는 학부모를 적대하지 않는다. 감정을 정제하고 표현을 다듬어 **학부모–교사 모두가 진짜 문제에 집중하도록 돕는 중간 다리**다.

이 시스템은 출품작(공모전)을 목표로 하며, "GPT 래퍼"가 아닌 **자체 학습 모델 + AI Hub 데이터 + 온디바이스 추론**으로 기술 깊이와 개인정보 안전성을 어필한다.

## 1. 왜 이 프로젝트인가 (문제 정의 — 발표 근거)

연구 결과 두 가설이 다출처 실측 데이터로 검증되었다.

- **가설 1 — 민원은 예측 가능한 패턴을 가진다 ✓**
  - 빅카인즈 7,957건/84개월 분석. ACF lag-1 = **+0.543**, STL 4월 계절성분 **+69.0건**, Kruskal-Wallis **p=0.003**.
  - 민원은 학사일정 분기점(**4·5·7·9·11월**: 평가·체험학습·생기부 마감·수학여행)에 반복 정점 → **사전 대응 시스템 설계가 정당화됨.**
- **가설 2 — 민원이 교육활동 축소를 만든다 ✓**
  - 교권보호위 심의 2019년 2,662건 → 2023년 **5,050건**(역대 최고). 서울 초등 숙박 수련회 21%(2023) → **3%**(2026).
  - 악성민원 → 교사 위축·형사책임 불안(89.6%) → 체험학습 소멸 → 학생 학습경험 빈곤화. → **시스템 도입의 사회적 당위성 확보.**

| 핵심 통계 | 수치 | 출처 |
|---|---|---|
| 악성민원으로 이직·퇴직 고려 | 64.2% | 전교조 경기지부 |
| 폭언·폭행 경험/목격 | 82.1% | 전교조 경기지부 |
| 침해 시 교사 혼자 대응 | 63.6% | 전교조 경기지부 |
| 사고 시 형사책임 불안 | 89.6% | 2026 전교조 |
| 악성민원 맞고소 의무화 찬성 | 97.7% | 교총 |

> 상세 근거·그래프·논문은 자료 레포 `학부모_악성민원_연구_종합.md`, `data/`, `그래프1~7_*.png` 참조.

## 2. 시스템 구조 (3단계) — 설계 다이어그램 기준

### 2-1. 전체 데이터 흐름 (다이어그램 1: 플로우차트)

```
[학부모가 민원 제기]
        ↓
[학부모 민원 수신/발송]
        ↓
[1단계: 실시간 민원 공격성 판단 엔진 가동]
   파인튜닝된 모델이 AI Hub 감성 대화 말뭉치 기준으로 공격성 점수 판단
        ↓
   ┌─────────────────────────────────┐
   │ 위험도가 일정 점수 이상인가?     │
   └─────────────────────────────────┘
        │ YES                      │ NO
        ▼                          ▼
   [악성/주의 민원 분류]      [일반 민원으로 분류]
        ↓
[2단계: AI Hub 공적 말하기로 파인튜닝된 로컬 생성형 LLM 가동 — Qwen2.5-14B(기본)/Ollama]
        ↓
   [감정 순화 필터 작동]  (+ RAG: 법률 DB 검색 → 교육청 지침 매칭)
        ↓
   [순화된 답변/요약 생성]
        ↓
[3단계: 담당 교사에게 전달 — 비동기 대시보드]
        ↓
[교사 답변 작성 → 실시간 린터 검증 → 학부모 발송 + 내부 기록 자동 저장]
```

### 2-2. 단계별 명세 (다이어그램 2: 표 — 입력/처리/예시)

| 단계 | 구성 요소 | 핵심 역할 (사용 데이터 / 기술) | 데이터 변환 예시 |
|------|-----------|-------------------------------|------------------|
| **1단계**<br>입력 및 분류 | **공격성 판단 엔진** | 감성 대화 말뭉치(AI Hub) 기반 모델로 학부모 민원의 **폭언·협박 수치 측정**. **정상 / 주의 / 위험 3클래스** 분류 모델 권장 | 입력: "애 관리를 어떻게 하길래 다쳐서 오냐, 당장 고소하겠다"<br>→ **(공격성 위험군 분류)** |
| **2단계**<br>정제 및 분석 | **감정 순화 필터 & RAG 엔진** | **Llama 3.x 기반**으로 감정을 제거하고, **법률 DB를 검색**하여 교육청 지침 매칭 | **[순화 요약]** "체험학습 중 발생한 아동 부상 경위 파악 및 조치 요구"<br>**[추천]** 관련 교육청 안전사고 지침 가이드 |
| **3단계**<br>출력 및 검증 | **비동기 대시보드 & 로컬 린터** | 수업 종료 후 일괄 전달. 교사가 답안 작성 시 **실시간 실수(과실 인정 등) 방지** | 교사 입력: "세심히 살피지 못해 죄송…"<br>→ 🚨 **Linter 경고: "사실관계 확인 중" 표현 권장** |

### 2-3. 각 단계 상세 기능

**1단계 — 공격성 판단 엔진 (격리/분류)**
- 입력 민원 텍스트를 **정상/주의/위험 3클래스**로 분류 + 공격성 점수화.
- 카테고리 분류: 안전·폭력 / 환불·분쟁 / 학습 상담 / 단순 문의 / 감정성 불만 / 칭찬·감사 (6종).
- **긴급 키워드 룰 별도**(학폭·아동학대·자해·고소 등) — 분류 결과와 무관하게 우선순위 상단 노출.
- 평가지표: **Macro-F1** + 혼동행렬. ★ **위험을 정상으로 놓치는 false negative 최소화가 최우선.**

**2단계 — 감정 순화 필터 + RAG**
- 공격성·과장·감정 표현 제거 → **핵심 요구만 공적 표현으로 변환**.
- **메타데이터 보존**: 반복성·강도·원어조를 분리 저장(가스라이팅/반복 민원 패턴 탐지).
- RAG: 법률·교육청 지침 DB 검색 → 근거 지침 매칭·추천.

**3단계 — 비동기 대시보드 + 답변 코칭 린터**
- **순화 카드 뷰 기본 + "원본 보기" 토글**(완전 은폐 아님).
- 우선순위 큐(긴급도순), 카테고리 필터.
- 교사 답변 작성 시 위험 표현(과실 인정·방어적·책임 회피·비교·반박) **실시간 빨간줄 + 대안 제안**.
- **"AI 제안, 최종 책임은 교사" disclaimer 고정.**

#### 답변 코칭 리스크 패턴 (3단계 린터 룰 시드)

| 패턴 | 예시 표현 | 코칭 메시지 |
|------|----------|-----------|
| 방어적 표현 | "그건 오해이십니다" | 학부모 부정 → 공감 후 설명 |
| 책임 회피 | "저희도 최선을 다했습니다" | 모호한 해명 → 구체적 사실 제시 |
| 비교·반박 | "다른 학부모님들은 만족하십니다" | 비교는 반발 유발 → 개별 사안 중심 |
| 즉답 회피 | "확인해 보겠습니다"만 반복 | 시간 약속·후속 조치 명시 |
| 과실 인정 | "세심히 살피지 못해 죄송…" | "사실관계 확인 중" 표현 권장 |

## 3. 확정된 핵심 설계 결정 (변경 금지 — 발표 방어 논리)

1. **순화 뷰 기본 + 원본 항상 열람 가능** — 정보 은폐가 아니라 우선순위 변경. (완전 은폐는 법적·윤리적 리스크라 폐기)
2. **긴급 분기** — 안전·법적 사안(학폭·아동학대·자해·고소)은 룰로 즉시 위험 상단 노출. 진짜 긴급 민원 필터링 사고 방지.
3. **로컬 추론(외부 API 미사용)** — 학부모 민원은 개인정보. 외부 API 전송 시 개인정보보호법 이슈. 공모전에서 "자체 학습 모델" > "API 래퍼" 점수 우위.
4. **메타데이터 보존** — 반복성·강도·원어조 분리 저장.
5. **AI 제안 disclaimer 고정** — "최종 책임은 교사" UI 상시 표시.
6. **MVP 범위 = Full 3-Layer 전부 구현** — 시연 임팩트 우선.
7. **합성 데이터 명시** — 학습 데이터에 실제 학부모 발화 없음(가상 시나리오 자체 작성). 발표 시 명시 = 가산점.

## 4. 기술 스택

### 환경 / 운영
- **모델 학습**: 맥북 M5 Max 128GB (Apple Silicon). 파인튜닝은 Colab Pro(A100/L4) 병행.
- **코딩 동기화**: GitHub Desktop으로 윈도우 ↔ 맥 동기화. 무거운 학습 산출물은 git 제외, 코드/노트북 골격만 동기화.
- **추론**: 로컬. **Ollama 확정** (GGUF 디코더 LLM 서빙). MLX는 선택적 대안.

### 모델 선택 (✅ 확정)

> **핵심 원칙: 모델은 2개. 분류기(인코더)와 순화 생성기(디코더)는 별개다.**
> KoBERT는 인코더 전용(MLM)이라 **분류·점수화만** 가능하고 **문장 생성(순화 rewrite)은 불가**하다. 따라서 2단계 순화는 반드시 별도의 **생성형 디코더 LLM**으로 처리한다.

| 용도 | 모델 | 비고 |
|------|------|------|
| 1단계 공격성 분류 | **KoBERT 파인튜닝** ⭐ (확정) | `skt/kobert-base-v1` + `kobert_tokenizer` (또는 `monologg/kobert`). AI Hub 감성 대화 말뭉치. 인코더 전용 = 분류·점수화 담당. 여유 시 KoELECTRA와 F1 비교 = 가산점 |
| 2단계 순화 변환 | **로컬 생성형 LLM — Qwen2.5-14B** + QLoRA (기본값, 확정) | **Ollama로 서빙** (`ollama pull qwen2.5:14b`). 한국어 순화 품질 기준 기본값. 대안: Llama 3.x (`ollama pull llama3.1`) — 둘 다 GGUF 디코더라 교체 용이. ⚠️ KoBART/KoT5는 Ollama 미지원(HF 전용)이라 Ollama 경로에서는 제외 |
| 3단계 답변 보조 | 2단계 모델 + RAG | 32B까지 맥북 감당 가능 |
| 양자화 | Q6_K 또는 Q8_0 | 128GB RAM 활용, 품질 손실 최소 |
| 추론 런타임 | **Ollama** (확정) | 파인튜닝 결과(LoRA 병합) → GGUF 변환 → `ollama create`로 로컬 서빙 |

### 검색/QA 보조 (다이어그램 1 모델 비교 결론)
공공기관 민원 챗봇 핵심 요건: **정확성·안정성·대화 자연스러움·운영 효율성**.
- **KoBERT** — 안정성·범용성 높음 → 공공기관 장기 운영 유리 (1단계 분류 기반).
- **KoELECTRA** — 검색·QA(문서 QA), 대화 대응에 강점 → **RAG/문서 QA 모듈에 병행 적용** 시 효과적.
- **결론: KoBERT 기반 + KoELECTRA의 대화 친화 특성을 보완적으로 활용하는 혼합 전략.**

### 백엔드
- FastAPI (Python 3.11)
- PostgreSQL + pgvector (RAG)
- WebSocket (실시간 민원 푸시)

### 프론트엔드 (✅ 2026-06-14 Vite 웹으로 확정 — 사용자 확정)
- **Vite + React + TypeScript + Tailwind**
- React Query (서버 상태), Zustand (클라 상태)
- 3단계 답변: **실시간 빨간줄 제거 → "AI 제안 답변을 교사가 편집"** 방식으로 단순화
  (Expo도 검토했으나, 빨간줄이 빠지면서 무거운 에디터가 불필요 → 웹 우선이 가장 가볍고 시연 유리)
- 답변 린터(`draft/validate`)는 유지하되 인라인 밑줄이 아닌 보조 점검으로 사용
- 앱이 필요하면 웹 코드를 Capacitor로 래핑 가능

### ML / 파인튜닝
- transformers + PEFT + TRL (+ Unsloth)
- QLoRA 4-bit (r=16, alpha=32 기준선)

## 5. 데이터 전략

### 1단계 (공격성 분류) — KoBERT
- **AI Hub 감성 대화 말뭉치** (※ 미신청 → **가장 먼저 신청**, 승인 1~3일).
- 감정 라벨 → 3클래스 매핑:
  - 분노/혐오/공격 → **위험**
  - 불만/슬픔/불안 → **주의**
  - 중립/기쁨 → **정상**
- 8:1:1 (train/val/test) 분할.
- 긴급 키워드 룰 사전 별도(학폭·아동학대·자해·고소 등).

### 2단계 (순화 변환)
- **AI Hub "공적 말하기 실습 및 평가 데이터"** — 공적/격식 말하기 문체 학습 핵심 소스. (AI Hub 신청 시 함께)
- 학부모↔교사 직접 데이터 부족 → 위 데이터로 공적 문체 학습 + **합성 데이터(공격 민원 ↔ 공적 표현 쌍) 1000건 목표**로 보완.
- 수작업 검수 100건 = 골든셋.

### 3단계 (RAG 문서)
- 교원지위향상법, 교육공무원법, 학교폭력예방법.
- 교육부·시도교육청 민원 응대 매뉴얼, 교권보호위 사례집(공개분).
- 청크 분할 → 임베딩 → pgvector 저장.
- (자료 레포 `논문_자료/`에 교육활동보호 매뉴얼·운영계획 PDF 다수 있음 — RAG 시드로 활용)

### 데이터 윤리
- 모든 정량 데이터는 공공 공시자료/공개 집계 수치. 실제 민원 텍스트는 **개발용 가상 시나리오로 자체 작성**, 실제 학교 민원 수집·저장 안 함.

## 6. 모노레포 구조 & API 설계

### 폴더 구조 (Phase 0)
```
/ (개발 레포)
├── ml/          # KoBERT 분류기, 순화 LLM 파인튜닝, 노트북, 변환 스크립트
├── backend/     # FastAPI, 파이프라인, pgvector RAG, 린터 룰
├── frontend/    # Vite + React + TS 대시보드 (웹)
├── data/        # 학습용 CSV (대용량 원본은 복사 X, 필요분만)
├── docs/        # 이 핸드오프 문서 + 설계 메모
└── .gitignore   # 모델·대용량 데이터 제외
```

### FastAPI 엔드포인트
- `POST /api/complaints/ingest` — 민원 수신 → 1·2단계 파이프라인 실행
- `GET  /api/complaints` — 대시보드 목록 (우선순위 정렬)
- `GET  /api/complaints/{id}` — 순화본 + 원본 + 메타데이터
- `POST /api/draft/suggest` — RAG + LLM 답변 제안
- `POST /api/draft/validate` — 답변 위험 표현 검출 (린터)

### 인프라
- PostgreSQL + pgvector docker-compose
- 추론: FastAPI ──HTTP──▶ Ollama (localhost:11434)
- 린터 정규식 룰 사전 100~200개

## 7. 단계별 실행 계획 (Phase)

### Phase 0 — 준비 (1~2일)
- [ ] 모노레포 구조 + `.gitignore`
- [ ] Python 3.11 + Node 20 환경
- [ ] **AI Hub 데이터셋 신청** (가장 먼저, 승인 대기 동안 다른 작업)
  - 1단계용: 감성 대화 말뭉치
  - 2단계용: 공적 말하기 실습 및 평가 데이터
- [ ] 맥북에 MLX/Ollama 설치 + 베이스 모델 다운로드

### Phase 1 — KoBERT 공격성 분류기 (1단계) ⭐ 첫 코드 지점
- 1-1. 라벨 매핑 + 긴급 키워드 룰 사전
- 1-2. **더미 데이터로 도는 학습→예측 파이프라인 골격 먼저 완성** (AI Hub 승인 대기용)
- 1-3. 승인 후 실제 데이터로 KoBERT 파인튜닝 (맥북 2~4h)
- 1-4. 평가(Macro-F1) + 서빙(ONNX 또는 transformers)

### Phase 2 — 순화 + RAG (2단계)
- 합성 데이터 생성 → QLoRA 파인튜닝(4~6h, 1~2회 재학습 각오) → RAG 구축(pgvector)

### Phase 3 — 백엔드 파이프라인 (3~5일)
- FastAPI: 수신 → 분류 → 순화 → 저장 → 조회 → 답변 제안/검증
- 모델 병합 → MLX/GGUF(Q6_K) 변환. 벤치마크 목표: 첫 토큰 1초 이내, 30 tok/s 이상

### Phase 4 — 프론트엔드 비동기 대시보드 (3단계)
- 순화 카드 + 원본 토글 + 우선순위 큐 + Tiptap 실시간 빨간줄 + disclaimer 고정
- 데모 시드 5~10개

### Phase 5 — 통합·시연·발표 (1주)
- E2E 데모, 정확도 측정표(분류 F1 / 순화 평가 / RAG 적중률), 시연 영상, 발표자료

### 전체 일정
| 시나리오 | 소요 |
|---|---|
| 풀타임 1명 | 6~8주 |
| 학교/직장 병행 | 3~4개월 |
